## Generation Process

The content rhythm is generated by the orchestrator after all system designs
are complete:

1. Read all system files from `.gf/stages/02-system-design/systems/`
2. Extract Section 7 (Day1-Day7 table) from each system
3. Aggregate by day: for each day, collect all system contributions
4. Build the Daily Rhythm table by synthesizing per-day system contributions
   into a coherent daily experience target
5. Verify content supply constraints against the aggregated data
6. Verify consumability constraints against the aggregated data
7. Write CONTENT-RHYTHM.md to `.gf/stages/02-system-design/`

## Constraint Verification Rules

### Content Supply Constraints

These are hard constraints from the source specification. Violations must be
resolved before the quality gate can pass.

| Constraint | Source | How to Verify |
|-----------|--------|---------------|
| Days 1-3: >= 1 new change/day | Spec 5.2 | Count `new_mechanic` entries per day in System Contributions |
| Days 4-7: >= 2 phase changes | Spec 5.2 | Count entries tagged as chapter theme / combo / branch change |
| Newbie task chain closure | Spec 5.2 | Verify onboarding system's Day1-Day3 entries form complete chain |
| Daily task cycle repeatable | Spec 5.2 | Verify task system has `reuse_variation` entries from Day 2 onward |
| Milestone rewards >= 3 tiers | Spec 5.2 | Count distinct reward tiers across growth/reward systems |

### Consumability Constraints

These ensure the player has meaningful engagement every day.

| Constraint | Source | How to Verify |
|-----------|--------|---------------|
| Daily perceivable progress | Spec 5.3 | Each day has at least 1 `new_mechanic` or 1 `reuse_variation` |
| Daily short goal (5-15 min) | Spec 5.3 | Each day has at least 1 system contribution achievable in 5-15 min |
| Daily medium goal (same-day) | Spec 5.3 | Each day has at least 1 system contribution achievable within the day |

## Language Rules

- Overview paragraph: user's configured language
- Table content: user's configured language
- System IDs: always English (SYS-UPPER_SNAKE-NNN)
- Content type tags: always English (new_mechanic, reuse_variation)
- Constraint names: user's configured language
