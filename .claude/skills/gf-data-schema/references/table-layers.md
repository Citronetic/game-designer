# 6-Layer Table Architecture Reference

This reference defines the 6-layer classification system for all data tables.
The schema-generator agent reads this to assign each table to the correct layer
and to determine load timing and hot-update behavior.

Source: `game-implement-process/stage3/系统提示词_数据拆解与建表.md` section C1
and section H (configuration loading and hot-update).

---

## Layer Definitions

### Layer 1: core_config

**Description:** Main gameplay configuration tables that define the core
game objects and mechanics -- levels, monsters, skills, characters, quests,
items, and their relationships.

**Characteristics:**
- Read frequency: very high (every gameplay tick or interaction)
- Write frequency: never at runtime (config is static during play)
- Data source: authored by designers in spreadsheets/tools
- Scope: global (same for all players)

**Load timing:** startup -- loaded once when the game client starts
**Hot-update:** rarely -- only in emergency patches; changes require careful
validation because these tables define core gameplay balance

**Example tables:** `level_config`, `monster_config`, `skill_config`,
`hero_config`, `item_config`, `quest_config`

---

### Layer 2: constants

**Description:** Fixed game constants that parameterize formulas, thresholds,
and system caps. These are global parameters not tied to specific game objects.

**Characteristics:**
- Read frequency: moderate (when formulas are evaluated)
- Write frequency: never at runtime
- Data source: authored by designers, rarely changed
- Scope: global (same for all players)

**Load timing:** startup -- loaded once, cached for session
**Hot-update:** never -- constants are structural anchors; changing them
invalidates all balance work and cached calculations

**Example tables:** `game_constants`, `formula_params`, `system_caps`

---

### Layer 3: probability

**Description:** Random event tables defining drop rates, spawn chances,
gacha pools, and weighted random selections. These tables drive all RNG
outcomes in the game.

**Characteristics:**
- Read frequency: high (on every random event trigger)
- Write frequency: never at runtime (rates are configured, not computed)
- Data source: authored by designers, frequently tuned during balance
- Scope: global (same for all players), but may vary by event period

**Load timing:** startup or enter_level -- depends on whether rates vary by
level context
**Hot-update:** may be hot-updated for live events -- event-specific drop
tables can be swapped without client release

**Example tables:** `drop_table`, `spawn_table`, `gacha_pool`

---

### Layer 4: i18n

**Description:** Localization strings mapping display keys to translated text
in each supported language. Referenced by `name_key` and similar string fields
in all other tables.

**Characteristics:**
- Read frequency: very high (every UI render)
- Write frequency: never at runtime
- Data source: translated by localization team
- Scope: language-specific (one set per language)

**Load timing:** startup -- loaded for the player's selected language
**Hot-update:** yes -- language patches can be pushed without client release;
new translations for events or corrections can be hot-loaded

**Example tables:** `i18n_strings`, `i18n_assets`

---

### Layer 5: progress

**Description:** Player progression data tracking save state, achievements,
unlocks, and inventory. Unlike Layers 1-4 (which are config data), Layer 5
is player-specific runtime data.

**Characteristics:**
- Read frequency: high (on every progression check)
- Write frequency: high (on settlement, level complete, item gain)
- Data source: generated by gameplay, not authored
- Scope: per-player (each player has their own data)

**Load timing:** settlement -- read and written during game result processing
**Hot-update:** not applicable -- this is player data, not config data; it is
read/written through save system, not config loading

**Example tables:** `player_progress`, `player_inventory`, `achievement_log`

---

### Layer 6: periodic_rewards

**Description:** Time-based reward schedules for daily login, weekly
challenges, seasonal events, and limited-time promotions. These tables drive
live operations (live-ops) content rotation.

**Characteristics:**
- Read frequency: moderate (checked on login, daily reset, event triggers)
- Write frequency: never at runtime (schedules are configured)
- Data source: authored by live-ops team, frequently rotated
- Scope: global but time-bound (schedules have start/end dates)

**Load timing:** startup -- loaded and cached, checked against current time
**Hot-update:** frequently -- live-ops content is the primary use case for
hot-update; new reward schedules, event configurations, and seasonal content
are pushed regularly without client release

**Example tables:** `daily_login_rewards`, `weekly_challenge`, `event_schedule`,
`seasonal_rewards`

---

## Classification Criteria

Use this decision tree to assign a table to the correct layer:

```
Is this table player-specific runtime data (not authored config)?
  YES --> Layer 5: progress
  NO  --> continue

Does this table define time-bound reward schedules for live-ops?
  YES --> Layer 6: periodic_rewards
  NO  --> continue

Does this table contain localization strings?
  YES --> Layer 4: i18n
  NO  --> continue

Does this table define probability/weight values for random outcomes?
  YES --> Layer 3: probability
  NO  --> continue

Does this table define global constants/formula parameters (not tied to
specific game objects)?
  YES --> Layer 2: constants
  NO  --> Layer 1: core_config
```

**Additional classification signals:**

| Signal | Points to Layer |
|--------|----------------|
| Table has a player_id field | Layer 5 (progress) |
| Table has start_date / end_date fields | Layer 6 (periodic_rewards) |
| Table has language_code field | Layer 4 (i18n) |
| Table has weight / probability / rate fields | Layer 3 (probability) |
| Table has very few rows (< 20) of global params | Layer 2 (constants) |
| Table defines game objects (monsters, items, levels) | Layer 1 (core_config) |

## Hot-Update Matrix

| Layer | Hot-Update Support | Reload Timing | Cache Invalidation | Version Consistency |
|-------|-------------------|---------------|-------------------|---------------------|
| 1: core_config | Rare (emergency only) | Full restart or controlled reload | All dependent caches must flush | Must reload sub-tables atomically with main tables |
| 2: constants | Never | N/A | N/A | N/A |
| 3: probability | Event-specific tables only | On event boundary or level enter | Drop/spawn caches flush | Independent per event; main probability tables stay |
| 4: i18n | Yes | Immediate (next UI render cycle) | String cache flush | Language pack is self-contained; no cross-table dependency |
| 5: progress | N/A (player data) | N/A | N/A | Save version migration handles changes |
| 6: periodic_rewards | Frequently | On next daily reset or event check | Reward schedule cache flush | Schedule tables are independent; can update one without others |

### Hot-Update Rules

1. **Atomic reload** -- When a hot-updatable table is updated, all tables that
   reference it via foreign keys must be checked for consistency. If a sub-table
   references a main table, both must be at compatible versions.

2. **Fallback on failure** -- If hot-update loading fails (corrupted data,
   validation error), the game must fall back to the previously loaded version
   and log the error. Never serve partially-loaded hot-update data.

3. **Version stamp** -- Every hot-update delivery includes a version stamp.
   The client compares its current version against the delivered version and
   only reloads if the version is newer.

4. **Static tables are immutable at runtime** -- Layer 1 (core_config) and
   Layer 2 (constants) should not be hot-updated in normal operations. If an
   emergency hot-update is needed for Layer 1, it must go through a full
   validation pipeline before deployment.

5. **Cross-layer consistency** -- When Layer 3 (probability) is hot-updated
   for an event, ensure that any new item IDs or monster IDs referenced in the
   updated probability table exist in the currently loaded Layer 1 tables.
   Missing references trigger fallback to previous version.
