# 工作流总览与执行手册

适用范围：将“30 秒左右小游戏广告素材”转化为可开发、可配置、可验证的完整游戏方案。

本文档目标：说明整套工作流如何执行、每一步产出什么、上下游如何对齐、什么情况下必须打回。

---

## 1. 总体流程

```text
Stage1（广告理解 -> 策划总纲）
  -> Stage2（总纲扩展 -> 完整系统设计）
    -> Stage3A（基于系统设计 -> 数据拆解与建表）
      -> Stage3B（基于建表与输入包 -> 数值配置与调参）
        -> Stage4（美术/UIUX/技术需求并行产出）
```

核心原则：
- 必须按顺序推进，禁止跳步。
- 每步都有门禁；不通过就打回上一步。
- 下游不得凭空新增上游未定义的核心系统。

---

## 2. 各阶段职责与输入输出

### Stage1：广告理解与策划总纲

目标：
- 从广告素材提炼核心玩法和产品方向，形成可扩展的策划总纲。

输入：
- 广告视频（约 30 秒）。
- 模板：`stage1/策划总纲模板.md`。

输出：
- Stage1 策划总纲成稿（按模板章节完成）。
- 每条关键规则带 `规则ID` 和“数据承载类型”。

门禁：
- 第 3/4/5/6/7 章不得空泛。
- 关键规则必须可被 Stage2/Stage3 承接。

---

### Stage2：完整系统设计（系统策划）

目标：
- 把“广告玩法”扩展成“可玩一周”的完整产品系统。

输入：
- Stage1 成稿。
- 规范：`stage2/完整系统设计执行规范.md`。

输出：
- 系统清单与依赖。
- 每系统详细规则（含系统ID、规则ID、状态、入口、异常）。
- 一周内容节奏（Day1-Day7）。
- Stage2 -> Stage3A 承接包（建表前置）。
- Stage2 -> Stage3B 数值前置输入包（7B）。
- Stage2 -> 程序实现合同包（7C）。

门禁：
- 缺系统ID/规则ID -> 打回。
- 缺 7B（数值前置）或 7C（程序合同）-> 打回。

---

### Stage3A：数据拆解与建表（系统/数据策划）

目标：
- 基于 Stage2 详细系统，完成“结构级”数据设计，让程序可开始建模和读取层开发。

输入：
- Stage2 成稿（含 7A/7B/7C）。
- 提示词：`stage3/系统提示词_数据拆解与建表.md`（Stage3A 部分）。

输出：
- 系统映射矩阵（Stage2 -> 对象 -> 表）。
- 表分层架构（主配置/常量/概率/多语言/进度/周期奖励）。
- 字段级表定义（主键/外键/类型/默认值/校验）。
- 关系图、加载时机、热更与回滚策略。
- 程序合同：状态机、事件、规则优先级、公式、错误码、模块边界。

边界：
- Stage3A 不做最终平衡数值。

门禁：
- 无法追溯 Stage2 的表或规则 -> 打回。
- 缺程序合同级信息 -> 仅可建框架，不可完整开工。

---

### Stage3B：数值配置与调参（数值策划）

目标：
- 在 Stage3A 已冻结表结构上，填写数值、设计难度和经济节奏，并提供调参与回滚方案。

输入：
- Stage2 的 7B 数值前置输入包。
- Stage3A 冻结版本。
- 模板：`stage3/数值配置工作文档模板.md`。

输出：
- 数值目标矩阵（生命周期/阶段目标）。
- 关段难度参数投放。
- 经济产消与卡点配置。
- 商业化边界下的数值约束。
- 调参策略、验证阈值、回滚策略。

边界：
- Stage3B 不得改 Stage3A 表结构与主外键。

门禁：
- Stage2-7B 缺项或 Stage3A 未冻结 -> 不可开始。
- 无阈值与回滚条件 -> 不可发布。

---

### Stage4：并行需求文档产出

目标：
- 为制作团队并行提供可执行需求。

输入：
- Stage2 成稿（主输入）。
- Stage3 结果（用于数据/交互/技术细节对齐）。

输出：
- 美术需求：`stage4/系统提示词_美术需求文档.md`。
- UIUX 需求：`stage4/系统提示词_UIUX需求文档.md`。
- 技术需求（建议按相同标准补齐）。

门禁：
- 与 Stage2 系统规则冲突 -> 打回。
- 与 Stage3 数据/状态机合同冲突 -> 打回。

---

## 3. 角色分工建议

- 系统策划：负责 Stage1、Stage2 主体。
- 数据策划：负责 Stage3A（建表）。
- 数值策划：负责 Stage3B（填数与调参）。
- 程序：从 Stage3A 程序合同开始并行开发框架，配合 Stage3B 联调。
- 美术/UI：按 Stage4 文档并行制作。
- QA：按 Stage3/Stage4 验收用例建测试集。

---

## 4. 必须执行的门禁规则

1) 不允许“无ID文档”流入下游：
- Stage2 没有系统ID/规则ID，Stage3 不得开工。

2) 不允许“无映射建表”：
- Stage3A 每张表必须能回溯到 Stage2 规则。

3) 不允许“边做边改结构”：
- Stage3A 冻结后，Stage3B 不得改主外键。

4) 不允许“无回滚发布”：
- Stage3B 未提供阈值与回滚方案，不得进入发布流程。

5) 不允许“文档漂亮但不可实现”：
- 缺状态机/事件/公式/错误码合同，程序仅能搭骨架，不算可开工。

---

## 5. 实战执行顺序（建议）

1. 按 `stage1/策划总纲模板.md` 产出总纲并过门禁。
2. 按 `stage2/完整系统设计执行规范.md` 完整扩展并补齐 7A/7B/7C。
3. 用 `stage3/系统提示词_数据拆解与建表.md` 先跑 Stage3A 并冻结。
4. 用 `stage3/数值配置工作文档模板.md` 执行 Stage3B。
5. 并行执行 Stage4 美术/UIUX/技术需求文档。
6. 联调时以 Stage3 程序合同为唯一逻辑口径。

---

## 6. 常见失败模式与应对

- 失败模式：Stage2 写得很长但没有规则ID。
  - 应对：先补 ID，再进入 Stage3。

- 失败模式：Stage3A 和 Stage3B 混写，表结构反复改。
  - 应对：先冻结结构，再填数值。

- 失败模式：数值只看难度，不看商业化和生命周期。
  - 应对：强制使用 Stage2-7B 输入包。

- 失败模式：程序拿文档还要二次开会猜规则。
  - 应对：补齐状态机、事件、公式、错误码、边界合同。

---

## 7. 最终交付判断

满足以下条件才可判定“工作流跑通”：
- Stage1 可复述核心玩法且规则可承接。
- Stage2 已形成完整系统设计并补齐 7A/7B/7C。
- Stage3A 已冻结表结构并给出程序合同。
- Stage3B 已形成可验证数值版本并具备回滚策略。
- Stage4 文档与 Stage2/Stage3 一致且可执行。

若任一条件未满足，必须回退对应阶段重做。
