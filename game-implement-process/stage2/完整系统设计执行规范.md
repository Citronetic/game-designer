# Stage2 完整系统设计执行规范

适用范围：本规范用于工作流第 2 步，将 Stage1 产出的策划大纲扩展为可开发、可拆表、可并行协作的完整系统设计。

上游输入：
- 必须以 `stage1` 中 LLM 已完成的策划案为唯一主输入。
- 该策划案必须按 `stage1/策划总纲模板.md` 结构输出。
- 若 Stage1 缺章（尤其第 3/4/5/6/7 章），Stage2 必须先补齐再扩展，不得跳步。

下游输出去向：
- Stage3A：数据拆解与建表（系统/数据策划工作）。
- Stage3B：数值配置与调参（数值策划工作）。
- Stage4：美术需求文档、UIUX 需求文档、技术需求文档并行产出。

---

## 0. Stage2 目标与最低交付标准

### 0.1 核心目标
- 将“30 秒广告玩法”扩展为“可连续游玩一周”的完整产品系统。
- 产出必须能被程序、策划、美术、UI、测试直接承接。

### 0.2 最低交付标准（硬约束）
- 文档必须覆盖完整系统清单，禁止只写玩法主循环。
- 必须包含一周内容供给与体验节奏（Day1-Day7）。
- 每个系统必须包含规则、状态、入口、出口、异常、依赖。
- 每个系统必须包含对下游的四类锚点：数据锚点、美术锚点、UI锚点、技术锚点。
- 全文禁止空泛句（如“增强体验”“更有趣”）且无机制解释。
- 每个系统必须带 `系统ID`，每条关键规则必须带 `规则ID`，否则 Stage3 不可承接。
- 必须包含“数值前置输入包”，否则 Stage3B 不得启动。

### 0.3 输出形态
- 主文档：`Stage2_完整系统设计.md`（建议命名，可按项目改名）。
- 附件表：
  - 一周体验节奏表
  - 系统依赖图
  - 内容投放表
  - 成长与任务投放表

## 1. 输入校验（进入 Stage2 前必须通过）

| 校验项 | 通过标准 | 不通过处理 |
|---|---|---|
| 上游来源 | 明确引用 Stage1 成稿路径 | 退回补全来源 |
| 核心玩法 | Stage1 第3章可复述单局流程 | 先补单局流程 |
| 系统循环 | Stage1 第4章有完整循环链路 | 先补循环定义 |
| 关卡内容 | Stage1 第5章至少有前30关节奏思路 | 先补关卡骨架 |
| 数值难度 | Stage1 第6章有阶段目标区间 | 先补阶段目标 |
| 前期体验 | Stage1 第7章有前10分钟节奏 | 先补新手体验 |

只要有 1 项不通过，禁止进入系统扩展。

## 3. 必做系统清单（最小集合，不可删减）

以下系统必须全部出现：
- 核心玩法系统
- 关卡与章节系统
- 机制/障碍/事件系统
- 结算与评分系统
- 资源与奖励系统
- 局外成长系统（至少 2 条成长线）
- 任务系统（新手任务、日常任务、阶段任务）
- 新手引导与前期体验系统
- 广告系统（激励与插屏）
- 轻付费与商店系统
- 解锁与进度门槛系统
- 存档与重进恢复规则系统

系统分层要求（新增）：
- 核心必做层（必须全部具备）：
  - 关卡内容系统
  - 对象/怪物系统
  - 角色成长系统
  - 资源与奖励系统
  - 任务与目标系统
  - 常量参数系统
  - 章节/地图推进系统
- 条件启用层（按项目边界决定）：
  - 轻付费与商店系统
  - 复杂广告策略系统
  - 长线活动系统

说明：
- 允许轻量项目采用“少系统 + 强参数化”，但核心必做层不可缺。

可选增强系统（按玩法补充）：
- 收集图鉴系统
- 装备/技能分支系统
- 随机事件系统
- 挑战模式系统

## 4. 单系统详细设计模板（每个系统必须完整填写）

每个系统必须按以下字段输出：

### 4.1 系统基础信息
- 系统名称
- 系统ID（唯一）
- 系统类型（核心玩法/成长/商业化/支撑）
- 优先级（P0/P1/P2）

### 4.2 系统目标
- 玩家价值目标（玩家为什么需要它）
- 业务目标（它支撑什么阶段目标）
- 成功判定（可观察结果，不写空话）

### 4.3 入口与触发
- 玩家从哪里进入
- 解锁条件
- 触发频次
- 首次触达时机

### 4.4 状态与流程
- 核心状态列表（至少 3 个）
- 状态切换条件
- 终止条件
- 失败回退路径

### 4.5 核心规则
- 触发条件
- 执行逻辑
- 输出反馈
- 异常处理

要求：每系统至少 5 条可执行规则，能被程序翻译为状态机或条件逻辑。
补充要求（必须）：
- 每条规则必须有 `规则ID`，格式：`RULE_系统ID_序号`。
- 每条规则必须标注 `实现类型`：配置驱动 / 逻辑驱动 / 混合驱动。
- 配置驱动或混合驱动规则，必须写明未来由哪类表承载。

### 4.6 系统依赖
- 上游输入系统与输入内容
- 下游输出系统与输出内容
- 强依赖/弱依赖标记
- 依赖断裂时兜底策略

### 4.7 一周内容供给贡献
- Day1-Day7 本系统每天提供什么新目标或新变化
- 内容是“新机制”还是“复用变化”必须标注
- 不允许连续两天完全无新内容贡献

### 4.8 下游承接锚点（必须）
- 数据锚点：未来需要哪些配置表（仅列表名和用途）
- 美术锚点：需要哪些资产和状态表现
- UI锚点：需要哪些页面、弹窗、交互组件
- 技术锚点：需要哪些能力（配置加载、结算、回调、容错）

新增“内容表达模型锚点”（必须）：
- 关卡数据组织方式：单行关卡 / 主从表关卡 / 节点式关卡。
- 空间表达方式：网格 / 坐标 / 角度半径 / 路径点。
- 事件表达方式：固定脚本 / 条件触发 / 波次触发。
- 复合字段约定：如数组分隔符、多段配置分隔符。

### 4.9 验收标准
- 功能完成判定
- 体验完成判定
- 可配置化判定
- 异常可恢复判定

### 4.10 程序承接四件套（新增，必须）
- 状态机草图：状态集合、状态切换条件、终止状态。
- 事件清单：事件ID、触发时机、事件载荷（payload）字段。
- 规则优先级：同帧或同节点冲突时的执行顺序。
- 异常补偿：失败场景下的补发、回滚、重试策略。

要求：
- 每个系统都必须提供 4.10，且必须可被 Stage3 直接引用。
- 若 4.10 缺失，判定该系统“不可开发”。

## 5. 一周可玩约束（Stage2 默认底线）

### 5.1 一周体验节奏表（必须）
| 天数 | 核心目标 | 新内容/新变化 | 成长变化 | 任务驱动 | 预期时长区间 |
|---|---|---|---|---|---|

### 5.2 内容供给硬约束
- 前 3 天：每天至少 1 个显式新变化（机制/障碍/目标类型其一）。
- 第 4-7 天：至少 2 次阶段性变化（章节主题、玩法组合、成长分支其一）。
- 一周内必须出现：
  - 新手任务链完整闭环
  - 日常任务循环可重复
  - 阶段奖励至少 3 档

### 5.3 可消费性约束
- 玩家每天都能获得“可感知推进”，不能只靠重复刷同关。
- 每天至少有一个“短目标（5-15分钟）”和一个“中目标（当日可达）”。

---

## 6. 跨系统一致性规则

必须显式检查并写明：
- 任务条件是否引用已开放内容（禁止引用未开放关卡/机制）。
- 成长消耗与资源产出是否匹配（禁止必然断流）。
- 广告补给与付费权益是否冲突（禁止互相抵消价值）。
- 新手引导是否覆盖核心规则（禁止核心规则未教先考）。
- 关卡压力点与补救机制是否成对出现。

## 7. Stage2 -> Stage3 数据前置要求（必须输出）

本章用于让 Stage3 直接开始建表，必须包含：
- 每个系统的“数据对象清单”（对象名 + 业务含义）。
- 每个系统的“候选表清单”（表名 + 用途 + 主关联对象）。
- 每条关键规则的“数据化落点”（配置驱动/逻辑驱动）。
- 全局枚举与ID前缀草案（避免后续命名混乱）。

新增必须项：
- 系统映射矩阵：`系统ID -> 对象 -> 候选主表/子表 -> 规则ID`。
- 复合字段编码草案：至少明确数组、二维坐标、多段配置的分隔规则。
- 表分层草案：主配置表 / 常量表 / 概率表 / 多语言表 / 周期奖励表 / 进度表。

限制：
- Stage2 只定义“要建什么表和为什么”，不做字段级建表。
- 字段级定义由 Stage3 执行。

## 7A. Stage2 -> Stage3A（建表）承接包

本节用于数据拆解与建表，必须输出：
- 系统映射矩阵：`系统ID -> 对象 -> 候选主表/子表 -> 规则ID`。
- 表分层草案：主配置表 / 常量表 / 概率表 / 多语言表 / 周期奖励表 / 进度表。
- 复合字段编码草案：数组、多段配置、坐标/路径的分隔规则。
- 配置与逻辑边界：哪些规则必须配置化，哪些必须代码化。

## 7B. Stage2 -> Stage3B（数值配置）前置输入包（新增，必须）

本节用于数值策划填表与调参，必须输出以下 8 类输入：

1) 生命周期目标矩阵：
- Day1-Day7 每日目标体验（新鲜感/挑战感/成长感）
- 阶段目标（教学期/建立期/压力期/回收期）

2) 关卡与节奏锚点：
- 关键关段划分（如 1-10, 11-30, 31-60...）
- 每段预期压力来源与恢复手段

3) 商业化约束：
- 广告介入边界（可介入节点/禁止打断节点）
- 轻付费介入边界（首触达时机、冲突回避规则）

4) 经济系统约束：
- 核心资源来源与用途
- 允许卡点与禁止断流场景
- 保底机制原则（失败补偿、低谷补给）

5) 数值控制杆清单：
- 关键可调参数（通关压力、产出、消耗、成长倍率）
- 参数影响面（影响哪些系统和体验）

6) 热更与调参权限：
- 可热更参数
- 非热更参数
- 风险参数（改动需审批）

7) 验证口径：
- 需要观察的核心指标（首通、重试、资源断流、任务完成等）
- 异常判定阈值（何时判定太难/太水/断流）

8) 数值禁区：
- 不允许通过纯数值硬拉解决的问题（应回到机制/关卡处理）

要求：
- 以上 8 类缺任一类，Stage3B 一律判定“输入不完整”。

## 7C. Stage2 -> 程序实现合同包（新增，必须）

本节用于保证程序可直接进入业务逻辑开发，必须输出：
- 系统状态机清单：`系统ID -> 状态枚举 -> 转移条件 -> 终止条件`。
- 事件总线清单：`事件ID -> 触发方 -> 消费方 -> payload`。
- 规则冲突优先级清单：同节点冲突时的排序与中断规则。
- 公式草案：关键计算项的伪代码表达（如伤害、评分、奖励、掉落）。
- 异常与错误码草案：错误类别、用户可见文案、系统处理动作。
- 客户端/服务端职责边界：谁计算、谁存档、谁校验、谁补偿。

要求：
- 若 7C 缺失，Stage3 即使建表完成也不得判定“可完整开工”。

## 8. Stage2 -> Stage4 并行需求前置（必须输出）

### 8.1 美术需求前置
- 资产清单草案（角色、敌人、机关、场景、道具、特效、UI图标）。
- 每类资产的状态需求（待机/受击/死亡/升级/禁用等）。
- 章节主题变化点（不止换皮，需对应机制变化）。

### 8.2 UIUX 需求前置
- 页面流（主界面、关卡、结算、任务、成长、商店、广告弹窗）。
- 核心交互（主按钮、二级按钮、关键反馈、错误反馈）。
- 广告与付费触点交互规则（出现节点、确认、失败兜底）。

### 8.3 技术需求前置
- 系统模块边界
- 配置加载时机
- 关键回调点（结算、广告、任务完成、奖励到账）
- 异常容错场景（广告失败、存档异常、配置缺失）

---

## 9. 质量门禁（Stage2 结束判定）

### 9.1 必过项
- 系统清单完整且每系统按模板填满。
- 一周体验节奏表完整，且每天有可感知推进。
- 已明确 Stage3 的数据前置清单。
- 已明确 Stage4 的三类并行需求前置。
- 已提供 `系统ID` 与 `规则ID`，且能追溯到 Stage1 的关键规则。
- 已说明内容表达模型，程序可判断是单表还是主从表承载。
- 已输出 Stage3B 数值前置输入包（7B 全部项）。
- 已输出程序实现合同包（7C 全部项）。

### 9.2 打回条件（任一即打回）
- 出现无规则支撑的口号式描述。
- 缺少任务或成长系统导致“一周可玩”无法成立。
- 依赖关系不清，无法判断开发先后。
- 无法回答“某系统失败时玩家看到什么、怎么兜底”。
- 无系统ID或无规则ID，导致 Stage3 无法映射承接。
- 未给出内容表达模型，导致 Stage3 只能拍脑袋建表。
- 未给出数值前置输入包，导致 Stage3B 只能主观猜数。
- 未给出程序实现合同包，导致程序只能二次会议反复确认。

---

## 10. 交付自检清单

- 是否明确标注 Stage1 输入文档路径与版本？
- 是否覆盖了最小必做系统清单？
- 是否给出 Day1-Day7 的内容与目标分配？
- 是否每个系统都给了数据/美术/UI/技术四类锚点？
- 是否产出可直接交给 Stage3 的数据前置说明？
- 是否产出可直接交给 Stage3B 的数值前置输入包（7B 全项）？
- 是否产出可直接交给程序的实现合同包（7C 全项）？
- 是否产出可直接交给 Stage4 的并行需求前置？

若有任意一项为“否”，不得进入 Stage3 或 Stage4。
